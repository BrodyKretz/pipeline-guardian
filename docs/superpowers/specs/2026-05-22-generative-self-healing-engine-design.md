# Generative Self-Healing Engine — Design (Phase 1)

**Date:** 2026-05-22
**Status:** Approved (pending spec review)
**Scope:** Phase 1 of a larger effort. Phases 2 (live file-visualization UI) and 3
(custom dataset ingestion) are deferred and sketched at the end.

## Problem

Today, "AI mode" is AI in name only. The chaos agent picks from a fixed list of
7 sabotages and applies a hardcoded mutation. The diagnosis agent calls one
`inspect_data` tool and returns a label from that same 7-item enum. The patch
agent **ignores the LLM entirely** and runs hardcoded string-replacements keyed
by the enum (`agents/patch_agent.py`). The "healing" is either a baseline file
copy or a canned code patch. Swapping the LLM out changes nothing.

We want AI mode to be genuinely generative end-to-end: no enums, no hardcoded
heals. The chaos agent invents its own breaking changes; the healing agents read
the real files, reason about what broke, and write the fix themselves — the way a
real on-call engineer would.

## Goals

- In AI mode, **zero hardcoded heals and zero failure-type enum**. Every repair is
  generated by the model.
- Chaos is **creative but bounded** — it invents one small breaking change per
  incident, to either the data file or the pipeline code, without being able to
  damage the harness itself.
- Healing agents **read the actual files and reason** to diagnose and fix.
- Emit a faithful, attributed **file-change event stream** so a later UI phase can
  visualize damage and repair live. The AI is blind to this stream.
- **Mock mode is unchanged** — the deterministic enum path stays so the project
  still runs free and offline.

## Non-Goals (this phase)

- The file-visualization dashboard UI (Phase 2).
- Custom/arbitrary dataset ingestion and auto-derived compliance profiles (Phase 3).
- Changing the scheduler, control endpoints, or incident persistence beyond what
  the new event/agent shapes require.

## Core principle: two things the AI is blind to

The whole design hangs on a clean separation between what the agents perceive and
what is harness-only ground truth:

1. **The baseline** (`baseline/`). Used only by the reporter to reset both working
   files between incidents so each round starts clean. Healing agents never read
   it — it is not an answer key.
2. **The colors / damage-vs-heal attribution.** Computed by the harness for the
   human watching, attached to file-change events. Agents never receive these
   events; they only ever see raw file text via their read tools.

This mirror — baseline and color attribution are both invisible to the AI — is the
spine of the design. The model must *find* the problem in plain text, exactly like
an engineer who wasn't told what changed.

## Mode behavior

- `llm.USE_REAL == False` (mock): existing deterministic path is preserved
  verbatim. Chaos picks from `SABOTAGE_TYPES`; diagnosis uses `_classify`; patch
  uses `_apply_fix`. No change.
- `llm.USE_REAL == True` (AI): the generative path below runs. The enum, the
  rule-based classifier, and the hardcoded patch table are not consulted.

## Architecture: agentic tool-use loops

Each AI agent runs a Claude tool-use loop (reusing `_anthropic_tool_loop` in
`llm.py`). Agents are given read tools and, where they act, write tools. This makes
"the agent reads the pipeline, reads the file, and diagnoses" literally true and
lets the patch agent iterate (write → test-run → retry).

Shared tools (read-only, available to the reasoning agents):

- `read_data()` → raw text of `data/weather_source.json`
- `read_pipeline()` → raw text of `pipeline.py`
- `dry_run_pipeline()` → runs the pipeline in isolation and returns the structured
  result (success, rows, error) **without** persisting output

None of these expose the baseline or any color/attribution metadata.

### Chaos agent (creative, bounded)

In AI mode, `run_chaos`:

- Reads the current `data/weather_source.json` and `pipeline.py`.
- Is instructed (system prompt) to act as a chaos engineer and introduce **one
  small, plausible breaking change** to **exactly one** of the two files — schema
  drift, type corruption, a subtle code bug, a flipped constant, etc. — that would
  cause the ETL to fail or emit bad data.
- Writes the new full file content back via the instrumented write helper (see
  File-change event model), tagged `kind="damage"`.
- Records a short free-text note of its intent for the incident log only. **This
  note is never passed to the healing agents.**

Boundedness / anti-"outwit" guardrails:

- **File whitelist:** the write tool refuses any path other than
  `data/weather_source.json` and `pipeline.py`. The baseline, `llm.py`, `main.py`,
  `agents/`, `config.py`, `restore.py` are unreachable.
- **One file per incident.**
- "Small tweak" framing in the prompt.
- The reporter restores the baseline at incident close, so no damage accumulates
  across rounds.

### Diagnosis agent (free-form)

Replaces enum classification. Given the pipeline error plus its read tools, it
investigates and returns:

```
{ "root_cause": <free text>, "approach": <free text plan>, "confidence": <0..1> }
```

No `failure_type` enum. Low confidence still routes to escalation (existing
`CONF_UNVERIFIED` gate stays as a confidence threshold, just applied to free-form
output).

### Patch agent (generative, no hardcoded heals)

Given the diagnosis and its read/write tools (but **not** the baseline):

- Reads the current files, reasons, and writes fixes to `data/weather_source.json`
  and/or `pipeline.py` via the instrumented write helper, tagged `kind="heal"`.
- May call `dry_run_pipeline()` to test its own fix and iterate within the loop.
- Returns a free-text description of what it changed.

`_apply_fix` and the `_SCHEMA_OLD/_NEW` constants are removed from the AI path.

### Validator agent (AI judge)

Re-runs the pipeline, reads the output, and **judges** whether the result is clean,
internally-consistent data (uniform keys, consistent types, sane row count, no
nulls) → `{ passed: bool, reasoning: <free text> }`. No hardcoded `_EXPECTED_KEYS`
schema in the AI path.

### Reporter agent

Flow unchanged. Logs the free-text diagnosis and fix (instead of enum + canned fix
string), updates the summary, then `restore_all()` resets both working files from
baseline for a clean next round.

## Failure handling: escalate with one retry

When validation fails (or the patch errors / times out):

1. The validator's failure `reasoning` is fed back to the patch agent for **one**
   fresh attempt.
2. If the retry still fails, the incident is marked **escalated** and logged. The
   working files are left in their damaged state until the reporter's
   end-of-incident baseline restore. (Phase 2: the unhealed regions stay red on
   screen for the human.)

## Execution safety: subprocess isolation

AI-generated `pipeline.py` is now untrusted code that we reload and execute. To
prevent a generated infinite loop or hang from wedging the monitor/scheduler:

- The pipeline runs in a **separate subprocess with a hard timeout** (e.g. 10s) and
  is killed on overrun; the structured result is serialized back across the
  boundary.
- A timeout/crash is surfaced to the agents as an ordinary failure result (so
  diagnosis/patch can react), not an exception that kills the chain.

Chosen over a thread+timeout because it fully isolates the server process from
arbitrary generated code — the production-minded choice, with no demo downside.

## File-change event model (the Phase 1 ↔ Phase 2 seam)

A single instrumented write helper is the only way any agent mutates a working
file. On each write it:

- Captures `before` (prior file content, or `null` if absent) and `after`.
- Emits a `FILE_CHANGED` event:
  ```
  { path, agent, kind: "damage" | "heal" | "restore", before, after, timestamp }
  ```
- `kind` is set by the caller: chaos → `damage`, patch → `heal`, reporter's
  baseline reset → `restore`.

This event flows over the existing `EventBus` / WebSocket bridge to the dashboard.
**It is never returned to any AI agent** — agents read files only through the
read tools, which return raw text with no attribution. Phase 2 will diff
`before`/`after` and color changed regions (damage → red; heal → the healing
agent's color; unhealed → stays red).

## Components touched

- `agents/chaos_agent.py` — AI branch: creative single-file sabotage via write
  helper.
- `agents/diagnosis_agent.py` / `llm.py:diagnose` — free-form diagnosis tool loop.
- `agents/patch_agent.py` / `llm.py` — generative patch tool loop; remove hardcoded
  heal table from the AI path.
- `agents/validator_agent.py` — AI-judge branch.
- `tools/pipeline_tools.py` — subprocess-isolated `run_pipeline`/`dry_run_pipeline`;
  instrumented write helper emitting `FILE_CHANGED`.
- `llm.py` — new tool definitions/executors for read/write/dry-run; chaos and
  validator AI branches.
- `state.py` — no shape change expected (events already free-form); confirm
  `FILE_CHANGED` rides the existing `emit` path.

## Testing strategy

- Mock-mode regression: existing `tests/` must pass unchanged (deterministic path
  untouched).
- Chaos write-whitelist: attempting to write a non-whitelisted path is rejected.
- File-change events: a chaos write emits one `FILE_CHANGED{kind:"damage"}` with
  correct `before`/`after`; a patch write emits `kind:"heal"`.
- AI-blindness: assert no read tool / agent input ever contains `kind`, color, or
  baseline content. (Static check on the tool executor outputs.)
- Subprocess safety: a pipeline that sleeps/loops is killed at the timeout and
  surfaced as a failure result, not a hang.
- AI agent loops: tested with a stubbed/mock Anthropic client returning canned
  tool-use sequences (no live API in CI), asserting the agent reads then writes and
  produces the expected event/record shapes.

## Deferred — future phases

- **Phase 2 — Live file-visualization UI.** A dedicated sub-tab/screen in the
  dashboard that renders every changed file's raw text and animates changes live:
  damage in red, heals in the healing agent's color, unhealed damage staying red.
  Any file an agent modifies is visualizable. Consumes the Phase 1 `FILE_CHANGED`
  stream. Colors remain a human-only overlay.
- **Phase 3 — Custom dataset ingestion + compliance profile.** Let the user drop in
  an arbitrary dataset. On ingest, the system profiles it and snapshots a
  "compliance standard" (structure/types/constraints) that becomes the correctness
  target: chaos deviates from it, healing restores compliance with it. **Open
  question to revisit then:** this reintroduces the "data contract" we set aside in
  favor of "infer from data alone" — decide whether the compliance profile is
  AI-visible (a contract the agents reason against) or stays harness-only ground
  truth like the baseline.
